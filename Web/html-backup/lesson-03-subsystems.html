It's week 5 of build season. Your robot has a drivetrain, an elevator, a scoop, and an arm. Every mechanism is controlled by code. If you put all that code in one file, that file is about 1,200 lines long. Two programmers are trying to edit it at the same time. Nobody is sure which lines control what. The robot match starts in 10 minutes.

This is the situation that subsystems exist to prevent.

Want the full picture of how a match loop ties subsystems together? See the Program Execution Flow reference.

Your brain is probably thinking this right now

"That sounds like a people problem, not a code problem. Can't everyone just be careful?"

They can try. They will fail. Code structure isn't about discipline - it's about making the right thing easy and the wrong thing hard. Subsystems make it structurally impossible to accidentally edit the elevator while you're debugging the scoop.

The cafeteria analogy

Imagine your school cafeteria. There's a pizza station, a salad bar, and a dessert section. Each one does its own job. The pizza station doesn't also handle dessert. If the pizza oven breaks, the ice cream is fine.

Your robot works the same way. Each subsystem owns exactly one mechanism and controls it completely. If the elevator breaks, the drivetrain keeps driving.

Drive

Moves the robot around the field. Owns the drive motors, encoders, and gyro.

Gets blamed for everything

Elevator

Lifts the scoop mechanism up and down. Owns the elevator motor and encoder.

Always needs one more centimeter of travel

Scoop

Tilts to retain or dump game pieces. Owns the scoop servo.

Deceptively simple. Servo angles are everything.

Arm

Rear arm for game piece control. Owns the arm servo.

The one mechanism everyone forgets to test first

Three reasons the structure earns its complexity

Reason 1 - One job, one place

If the scoop is broken, you look in one file. Not everywhere. When you're debugging at 2am before competition, "one file" feels like a gift from the universe.

Reason 2 - Teammates can work in parallel

Two people editing the same file is a merge conflict waiting to ruin someone's day. With subsystems, one programmer works on the elevator while another works on the drivetrain - different files, no conflict.

Reason 3 - WPILib enforces it

The CommandScheduler prevents two commands from using the same subsystem at the same time. If one command is driving, it won't let another command accidentally also try to drive. This prevents the robot from doing contradictory things simultaneously. It has better conflict resolution than most group projects.

What's inside a subsystem?

A subsystem is a Java class that extends SubsystemBase. Here's the anatomy - every subsystem has the same four parts:

1 - Fields

Hardware objects (motors, servos, encoders), goal state, current state. Declared at the top of the class. Private - nothing outside the subsystem touches them directly.

The subsystem's private possessions

2 - Constructor

Runs once when the robot starts. Configure hardware, set initial state, register with the scheduler. This is where motors get inverted, encoders get reset, and ports get assigned.

Robot startup checklist

3 - periodic()

Runs every 20ms, forever. Read sensors, log data, update internal state. No motor control here. Motor control goes in commands.

The heartbeat. Every loop. Always.

4 - Methods & Commands

Public methods that expose what the subsystem can do. Often return Command objects that callers can schedule. This is the subsystem's public API.

What the rest of the robot is allowed to ask for

Why periodic() cannot control motors

The CommandScheduler is the engine that runs your robot's commands - this is true on every command-based WPILib robot, XRP or full FRC competition bot, not just this lesson's example. Each 20ms loop it follows a precise sequence (documented at docs.wpilib.org): it calls every subsystem's periodic() first, then polls triggers and buttons, then runs active commands. The key insight: periodic() runs unconditionally - the scheduler calls it at step 1 every loop and has no way to skip, delay, or interrupt it. Commands are what the scheduler can start, stop, interrupt, and coordinate. Try it below - step through three scenarios and watch exactly what the scheduler does each loop.

If you put motor output in periodic(), you've placed it outside the scheduler's reach. A command can't override it. A disable can't stop it cleanly. Two things can try to drive the motor with no coordination. The simulator below shows what the scheduler actually sees in three scenarios.

What the three scenarios show

Good: periodic() reads sensors only. A drive command is scheduled, runs for several cycles, ends cleanly. The scheduler controls the full lifecycle.

Bad: Motor output is inside periodic(). The scheduler calls it at step 1 - unconditionally, every loop. It cannot skip it or cancel it. A robot disable has no effect on periodic(). The motor runs until power is cut.

Conflict: Two commands both require Drive. The scheduler detects this at step 3 and resolves it via getInterruptionBehavior() - either cancelling the old command (kCancelSelf, the default) or rejecting the new one (kCancelIncoming). No undefined behavior. This is the framework working correctly.

The key distinction: periodic() is invisible to the scheduler's resource management system. It is not a command. It has no requirements. The scheduler cannot prevent it from conflicting with a command. Putting motor output in periodic() bypasses every safety guarantee the framework provides.

periodic() vs commands - bad code / good code

@Override
public void periodic() {
    // ❌ WRONG - a beginner's natural instinct:
    // "periodic() runs every loop, so I'll read the joystick here"
    // Problem: scheduler calls this unconditionally at step 1.
    // No command can override it. No disable can stop it cleanly.
    double speed = controller.getLeftY(); // reads joystick
    leftMotor.set(speed);
    rightMotor.set(-speed);
}
@Override
public void periodic() {
    // ✅ CORRECT - sensors and logging only
    io.updateInputs(inputs);
    Logger.processInputs("Drive", inputs);
}

// Motor output lives in a Command - scheduler controls lifecycle
// In RobotContainer: drive.setDefaultCommand(drive.defaultDriveCommand(controller));
public Command defaultDriveCommand(XboxController controller) {
    return run(() -> {
        double speed = controller.getLeftY();
        io.setVoltage(speed * kMaxBatteryVoltage, -speed * kMaxBatteryVoltage);
    });
}

What a subsystem looks like in code

public class Scoop extends SubsystemBase {

    // All scoop code lives here.
    // Nothing else touches the scoop servo.
    // If someone adds drivetrain code here,
    // that person buys the team snacks.

    @Override
    public void periodic() {
        // Called at step 1 by the CommandScheduler every 20ms.
        // Read sensors, log data. No motor/servo output.
    }
}
Extending SubsystemBase registers this subsystem with the CommandScheduler automatically. That's the only requirement.

LoggedRobot, TimedRobot, and the command framework

Student
Is there a CommandRobot class we're supposed to extend?
Good question, common confusion
No. The inheritance chain is: your Robot extends LoggedRobot, which extends TimedRobot. The command framework isn't a base class - it's a system that runs because you call CommandScheduler.getInstance().run() in robotPeriodic(). That one line is what makes the whole framework go.
Student
So the command framework isn't built into the robot class at all?
Exactly
It's opt-in. You could have a TimedRobot with no commands at all - just raw periodic methods. Command-based is a pattern we choose to use by running the scheduler. AdvantageKit's LoggedRobot wraps each loop cycle to capture data. These are two separate systems that work together.
There are no dumb questions

Q: Can a subsystem control two mechanisms?

Depends what "two mechanisms" means. If two motors combine into one inseparable physical output - like a differential, where you can't even talk about "motor A's job" separately from "motor B's job," only the combined rotation and pitch they produce together - that's really one job done by two actuators. One subsystem, with the combine/inverse math inside it. But if two mechanisms are independently meaningful and just need to happen in sequence - like a climber and an elevator that extend, then deploy, then retract in order - keep them separate and coordinate with commands or a superstructure. The test isn't "do they move together," it's "can I describe what each motor does on its own?" If yes, separate subsystems. If the motors only make sense combined, one subsystem.

Q: What if two separate subsystems need to coordinate, like a scoop angle that depends on elevator height?

Ask yourself: does the logic belong in ElevatorSubsystem.periodic()? That works, but now Elevator knows about Scoop - tight coupling. Flip it - put it in ScoopSubsystem.periodic() instead? Same problem, just reversed. What if a third class owned both, and decided when each one should move? That's the superstructure pattern - a coordinator that sits above multiple subsystems without owning hardware itself. We're naming it here; building one comes later.

Q: How does a subsystem communicate with another subsystem?

Through commands in RobotContainer. Subsystems don't talk to each other directly - that would create dependencies that make debugging much harder. Commands coordinate between subsystems.

Q: What's the difference between periodic() and a command?

periodic() runs every loop cycle regardless - use it for reading sensors and logging. Commands run when scheduled - use them for actions triggered by buttons or autonomous. Sensor reading goes in periodic(). Motor control goes in commands.

The problem with importing hardware directly

Imagine this scenario

A subsystem has import edu.wpi.first.wpilibj.xrp.XRPServo at the top. You've been testing in simulation all week. Match day arrives - you plug into the real FRC robot (which uses a different servo controller). The class name is different. Your subsystem won't compile on the competition robot without changes. You are now editing code at a competition. This is bad.

The IO pattern solves this by adding a layer of indirection. Instead of the subsystem knowing which hardware object to use, you define an interface that describes what operations the hardware must support. Then you write separate implementations for each hardware type, including a simulation implementation with no real hardware at all. The subsystem only ever talks to the interface, so it runs identically whether it's driving a real XRP servo or a simulated one. That's what makes testing your code on a laptop, with no robot present, possible.

Four files instead of one

Arm.java - the subsystem
public class Arm extends SubsystemBase {

  // Talks to the interface - never to hardware
  private final ArmIO io;
  private final ArmIOInputsAutoLogged inputs
      = new ArmIOInputsAutoLogged();

  public Arm(ArmIO io) {
    this.io = io; // injected from outside
  }

  @Override
  public void periodic() {
    io.updateInputs(inputs);
    Logger.processInputs("Arm", inputs);
  }

  public void setAngle(double deg) {
    io.setAngle(deg); // through interface
  }
}
ArmIO.java - the interface
public interface ArmIO {

  // The data the hardware reads each loop
  @AutoLog
  class ArmIOInputs {
    public double commandedAngleDeg = 0.0;
  }

  // What any arm hardware implementation must do
  default void updateInputs(ArmIOInputs inputs) {}
  default void setAngle(double angleDeg) {}
}

And then a separate implementation for each hardware type:

ArmIOXRP.java - XRP hardware implementation
// This file knows about XRP hardware
import edu.wpi.first.wpilibj.xrp.XRPServo;

public class ArmIOXRP implements ArmIO {

  private final XRPServo servo = new XRPServo(5);

  @Override
  public void updateInputs(ArmIOInputs inputs) {
    inputs.commandedAngleDeg = servo.getAngle();
  }

  @Override
  public void setAngle(double deg) {
    servo.setAngle(deg);
  }
}
ArmIOSim.java - simulation stub (Optional)
// This file knows about simulation
// No hardware imports at all

public class ArmIOSim implements ArmIO {

  private double angleDeg = 0.0;

  @Override
  public void updateInputs(ArmIOInputs inputs) {
    inputs.commandedAngleDeg = angleDeg;
  }

  @Override
  public void setAngle(double deg) {
    angleDeg = deg; // just store it
  }
}

How RobotContainer wires it together

The subsystem never decides which implementation it gets. RobotContainer makes that decision at startup based on whether the code is running in simulation or on real hardware:

public RobotContainer() {
    switch (Constants.currentMode) {
        case REAL:
            arm = new Arm(new ArmIOXRP());  // real XRP servo
            break;
        case SIM:
            arm = new Arm(new ArmIOSim());   // simulation stub
            break;
        default:
            arm = new Arm(new ArmIO() {});   // log replay - no hardware
            break;
    }
}
The Arm subsystem itself never changes. Swap the implementation per case in RobotContainer. That's the entire point.

4
Add @AutoLogOutput so the goal appears in AdvantageScope
▶

@AutoLogOutput is an AdvantageKit annotation. Put it above a field and AdvantageKit automatically records that field to the log every loop cycle - no Logger.recordOutput() call needed. One annotation, instant visibility.

import edu.wpi.first.wpilibj.xrp.XRPServo;
import org.littletonrobotics.junction.AutoLogOutput;

/** Creates a new scoop. */
private final XRPServo scoopServo = new XRPServo(4);
@AutoLogOutput  // ← add this annotation
private Goal goal = Goal.FLAT;
After building and simulating, look for RealOutputs/Scoop/Goal in the AdvantageScope log tree. It'll show FLAT, CARRY, or DUMP in real time.
@AutoLogOutput vs Logger.recordOutput()

@AutoLogOutput works on fields - it logs the field's value every loop cycle automatically. Logger.recordOutput() is a manual call you write in code - useful for computed values or one-off outputs. Use @AutoLogOutput whenever possible; it's less code and harder to forget.

5
Add setGoal() and setGoalCommand()
▶

Add two methods: a simple setter, and a factory method that wraps it in a command. Commands are how the outside world controls the subsystem.

    /** Creates a new scoop. */
    public Scoop() {
        // Constructor - runs once at startup
    }

    // ← add this method - simple setter, changes the goal field
    public void setGoal(Goal newGoal) {
        goal = newGoal;
    }

    // ← add this method - returns a command that sets the goal once, then ends
    // runOnce() = run the lambda once, then the command is done
    public Command setGoalCommand(Goal newGoal) {
        return runOnce(() -> setGoal(newGoal));
    }

    // ← add this method - safe state, called when a command is interrupted
    public void stop() {
        setGoal(Goal.FLAT);  // return to resting position
    }

    @Override
    public void periodic() {
        // Called every 20ms (50Hz)
        // Sensors and logging only - no motor control
    }
runOnce() is a WPILib helper available inside SubsystemBase. It creates a command that automatically requires this subsystem, runs the lambda once, then ends. You don't need to call addRequirements(this) manually.
6
Wire periodic() to actually move the servo
▶

The goal is stored, but the servo isn't moving yet. periodic() reads the current goal and tells the servo where to go. This is the correct pattern: commands set the goal, periodic() applies it.

Wait - isn't this motor control in periodic()?

It looks like it, but this is actually the correct pattern for a servo. The command sets the goal (what position you want). periodic() reads that goal and applies it. The servo is always tracking the current goal - if the goal doesn't change, neither does the servo. This is different from leftMotor.set(0.5) in periodic(), which always pushes power regardless of any command state.

@Override
public void periodic() {
    // ← add this switch - apply current goal to servo
    switch (goal) {
        case FLAT  -> scoopServo.setAngle(0.0);
        case CARRY -> scoopServo.setAngle(45.0);

Working smarter - VS Code Snippets

You just typed a lot of boilerplate. Fields, enums, constructor, periodic, methods. You'll do this again for Elevator. And again for every subsystem on every future robot. Senior programmers don't retype this - they use snippets.

What is a VS Code snippet?

A snippet is a keyboard shortcut that expands into a block of code. Type aksubsystem, press Tab, and VS Code fills in the entire SubsystemBase shell with fields, constructor, and periodic already stubbed. You fill in the name and the specifics.

FRC teams that maintain snippet libraries - including teams like 6328 (AdvantageKit authors) - type substantially less boilerplate. The pattern is always the same; the details differ.

Creating the BearBots snippet file

Code diff challenge: which of these is the correct way to wire the motor?

@Override
public void periodic() {
    elevatorMotor.set(0.4); // always run at 40%
}
@Override
public void periodic() {
    switch (goal) {
        case STOWED -> elevatorMotor.set(0.0);
        case LOW    -> elevatorMotor.set(0.3);
        case HIGH   -> elevatorMotor.set(0.6);
    }
}
Option B is correct. Option A runs the motor forever at 40% regardless of goal state - the elevator will never stop. Option B checks the goal and applies the appropriate speed. When goal is STOWED, the motor is 0. Commands set the goal; periodic() applies it. This is the same pattern as the scoop servo.
Gold - design the full robot architecture

Before writing any code: design the complete subsystem structure for a new robot.

  • List every mechanism on the BearBots robot (real or extended-hypothetical)
  • For each subsystem: what hardware does it own? What are its goal states? What commands does it expose?
  • Identify coordination scenarios: the elevator rises → scoop must tilt up to retain the piece. Which commands coordinate these two subsystems? Where does that coordination live?
  • Create all subsystem shells using the aksubsystem snippet - no implementation yet
  • Verify the project builds with all shells registered in RobotContainer
  • Add a brief comment in each class describing what it owns and what commands it will eventually expose
Subsystems coordinate through commands, not direct method calls. The elevator rising while the scoop tilts is a command that requires both subsystems - it calls elevator.setGoal(HIGH) and scoop.setGoal(CARRY) in sequence or in parallel. The command lives in RobotContainer (or a dedicated auto/teleop command file), not inside either subsystem. That distinction is critical for scheduling: WPILib can interrupt either subsystem's default command when this coordinated command needs control.

Bug 1 - Scoop servo living inside Drive

It was convenient. It is not correct. "Convenient" and "correct" are different things that feel the same until 11pm before competition.

The scoop servo belongs in Scoop. When the scoop misbehaves, you'll search Drive for the bug. Worse: a command that tilts the scoop will accidentally require Drive - which means the robot can't drive while tilting the scoop. One misplaced field locks out an entire subsystem.

Fix
Move scoopServo to Scoop.java. You already built it in Part 4.

Bug 2 - Motor control in periodic()

The robot drives forward. Forever. Without stopping. Even in autonomous. Even when disabled. This is what "always" means in robot code.

Motor control in periodic() runs every loop cycle regardless of mode or state. The robot will drive forward the moment it's enabled and never stop. Motor control belongs in commands - which can be started, stopped, and interrupted. periodic() is for reading sensors and logging only.

Fix
Move motor control into a default drive command. Set it as the default command in RobotContainer. periodic() reads sensors and logs only.

Bug 3 - tiltScoop() requires Drive, not Scoop

The comment says "requires Drive - correct!" The comment is not correct.

Because tiltScoop() is defined inside Drive, WPILib's runOnce() automatically requires the drive subsystem. This means tilting the scoop locks out any drive commands - the robot can't drive while the scoop is tilting. Commands must live in the subsystem that owns the hardware they control.

Fix
Move tiltScoop() to Scoop. It will then correctly require only the scoop, and the drive subsystem stays free.

Bug 4 - emergencyStop() doesn't stop the real problem

You stopped the motors for one loop cycle. Then periodic() ran again on line 8 and set them back to 0.5. The emergency stop stopped for approximately 20 milliseconds.

Setting motors to 0 in a runOnce() command does nothing if periodic() is setting them to 0.5 every loop. The command runs once and ends - then periodic() immediately overrides it. The fix for an emergency stop is to fix Bug 2 first (remove motor control from periodic), then set a goal state to STOPPED that periodic() reads and applies. This is why motor control in periodic() creates a second-order problem: it makes emergency stops impossible without a full architecture fix.

Fix
Fix Bug 2 first. Then implement a proper STOPPED goal state that periodic() reads. The emergency stop command sets goal = STOPPED; periodic() applies 0 power when it sees that goal.
