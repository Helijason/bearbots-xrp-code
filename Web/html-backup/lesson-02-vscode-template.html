Git basics — your three commands

Git tracks changes in snapshots called commits. Think of each commit as a save point you can return to. Here are the three commands you'll use every session.

After you make changes to your code

Stage your changes — tell Git which files to include in the snapshot:

git add .

The . means "everything changed in this folder." You can also stage a single file: git add Robot.java

Commit — save the snapshot with a message describing what you did:

git commit -m "Add pilot name logging to Robot.java"

The message is for you and your teammates. Be specific — "fixed stuff" is useless at 11pm the night before competition.

Pull — get the latest updates from the shared repository:

git pull

Run this at the start of every session so you're working from the latest code.

What about pushing?

git push uploads your commits to GitHub so teammates can see them. You'll do this later once your changes are ready to share. For now, commit locally — your work is saved and tracked even without pushing.

Step 1 — What the XRP-TEMPLATE project generates

Look at the XRP-TEMPLATE project first. Open the Explorer panel (Ctrl+Shift+E) and expand src/main/java/frc/robot/. You should see exactly these files:

Main.java  click to expand

The entry point. Starts the robot.

Job: The very first thing that runs when the robot boots. Calls RobotBase.startRobot(Robot::new) which hands control to your Robot class.

Should you modify it? Almost never. Only touch this if you rename your Robot class.

The rule

The comment at the top of Main.java says "Do not add any static variables or initialization code here." It means it. WPILib needs to control startup order — anything added here runs before the hardware layer is ready.

Robot.java  click to expand

The lifecycle manager. Controls what happens in each robot mode.

Job: Contains lifecycle methods — robotPeriodic(), autonomousInit(), teleopInit(), etc. Called automatically by WPILib at the right times.

XRP-TEMPLATE gives you: extends TimedRobot. Notice that. You'll see why it changes in the comparison section below.

Most important line: CommandScheduler.getInstance().run() in robotPeriodic(). Without it, no commands ever execute. Ever.

@Override
public void robotPeriodic() {
    CommandScheduler.getInstance().run(); // don't remove this
}

RobotContainer.java  click to expand

The switchboard. Creates subsystems and wires up buttons.

Job: Instantiates subsystems, sets up controller bindings, provides the autonomous command. The main switchboard of the robot.

Why not put this in Robot.java? Separation of concerns. Robot.java handles timing. RobotContainer handles what the robot actually does. Both files stay readable.

This is the file you'll edit most. Every new subsystem and button binding lives here.

Constants.java  click to expand

A place for numbers that don't change. Starts nearly empty.

XRP-TEMPLATE gives you: an almost empty file with one comment: "place constants here." That's it.

The intent: motor IDs, speed limits, PID gains — anything that's a fixed value that might need changing later. Hardcode a motor ID in a subsystem and you'll search 6 files when wiring changes. Put it in Constants.java and you change it once.

Constants.java isn't the only constants file

On real competition robots, Constants.java gets crowded fast. Teams split it out:

What Where teams put it
Swerve offsets, gear ratios, PIDTunerConstants.java or SwerveConstants.java
Field layout, AprilTag positionsFieldConstants.java
Arm limits, speeds, gainsArmConstants.java or inner class in subsystem
CAN bus IDsConstants.java inner class, or CANConstants.java
Vision camera configsVisionConstants.java

There's no single right answer — this is a team design decision. Start in Constants.java. Split into separate files when it gets crowded. The BearBots project uses inner classes (Constants.DriveConstants, etc.) as a middle ground.

build.gradle  click to expand

The build recipe. Tells Java what libraries to include.

Job: Configures the build system. Specifies WPILib version, vendor libraries, compile and deploy settings.

When you add AdvantageKit via WPILib: Manage Vendor Libraries, it edits this file automatically. Rarely touched by hand.

If your build fails mysteriously — check here first. Mismatched versions or missing vendor deps are usually the culprit.

ExampleSubsystem.java + ExampleCommand.java  click to expand

Placeholder files. Gone in the BearBots project.

WPILib generates these as structural examples showing how a subsystem and command are shaped. They don't do anything real — no hardware, no logic worth keeping.

In the BearBots project these are deleted and replaced with DriveSubsystem.java and our actual robot files. You'll learn what goes inside subsystems in Lesson 3.

Step 2 — Compare XRP-TEMPLATE vs BearBots

Now open both projects side by side. The BearBots project isn't XRP-TEMPLATE — we made deliberate changes before you got here. Each card below shows one difference, why it was a problem, and why the change fixes it.

Your job

Find each change in the BearBots project before reading the card. The file name is labelled on every card. Spot the difference first — then click to see the reasoning.

Robot.java Class declaration
📄 XRP-TEMPLATE
public class Robot extends TimedRobot
🐻 BearBots
public class Robot extends LoggedRobot
⚠️ Problem this solves

TimedRobot has no logging hooks. Your robot breaks at competition on Saturday. You have zero recorded data to understand what happened. You go home guessing.

LoggedRobot extends TimedRobot — every lifecycle method works identically. The difference is that LoggedRobot wraps each 20ms loop cycle with AdvantageKit data capture. Every input the robot reads, every output it commands, timestamped and saved. You can replay the entire match in simulation later and see exactly what happened, when it happened.

All the same methods still exist: robotPeriodic(), teleopInit(), etc. Nothing breaks. You just gain observability.

Robot.java Constructor
📄 XRP-TEMPLATE
public Robot() {
  m_chooser.setDefaultOption(
    "Default Auto", kDefaultAuto);
  m_chooser.addOption(
    "My Auto", kCustomAuto);
  SmartDashboard.putData(
    "Auto choices", m_chooser);
}
🐻 BearBots
public Robot() {
  Logger.recordMetadata(
    "ProjectName", "XRP-AKit");
  Logger.addDataReceiver(
    new NT4Publisher());
  Logger.start();
  robotContainer =
    new RobotContainer();
}
⚠️ Problem this solves

AdvantageKit is installed but never told to start. Every @AutoLog annotation silently does nothing. No error, no warning — just no data, ever.

Logger.start() initializes all data sinks and starts the capture loop. It must happen before new RobotContainer() — subsystems are created inside RobotContainer, and we need logging active before their first periodic() call so no data is missed from boot.

Note: this is Logger.start() — not Logger.getInstance().start(). AdvantageKit 3.x changed the API. The old pattern still appears in many tutorials online. Ours is correct for 2025+.

Robot.java Logger configuration
📄 XRP-TEMPLATE
// no Logger config
// no data receivers
🐻 BearBots
Logger.recordMetadata(
  "ProjectName",
  "XRP-AKit");
Logger.addDataReceiver(
  new NT4Publisher());
Logger.start();
⚠️ Problem this solves

Without a data receiver, AdvantageKit has nowhere to send data. NT4Publisher streams everything to NetworkTables so AdvantageScope can see it live during simulation.

Logger.addDataReceiver(new NT4Publisher()) tells AdvantageKit to publish all logged values to NetworkTables. That's what AdvantageScope connects to. Without it, Logger.start() runs but nothing appears in AdvantageScope.

On a real robot you'd also add a file logger to save data for replay. For the XRP simulator, NT4Publisher is all we need.

Constants.java Mode enum + currentMode field
📄 XRP-TEMPLATE
// Constants.java
// does not exist
🐻 BearBots
public static final
  RobotMode currentMode =
  RobotMode.SIM;

public enum RobotMode {
  REAL, SIM, REPLAY
}
⚠️ Problem this solves

No way to tell the codebase which environment it's running in. Teams hardcode for real hardware and lose the ability to simulate, or hardcode for sim and can't deploy without changes.

One field controls the entire IO layer. Set currentMode = RobotMode.SIM and the codebase loads simulated hardware. Set it to RobotMode.REAL and it talks to real motors. Set it to RobotMode.REPLAY and it reads from a saved log file.

On a roboRIO, the code auto-detects REAL using RobotBase.isReal() so you never accidentally deploy sim code. In VS Code, you set SIM manually. This is covered in when we go deep on the IO pattern.

⚠️ Problem this solves

Magic numbers scattered across subsystem files. Motor channel hardcoded as 0 or 1 in multiple places. Wiring changes on build day. You hunt through files, miss one, and the robot drives in circles at your first match.

Inner classes group related constants together without needing separate files. Constants.DriveHardware.kLeftMotorChannel is unambiguous about what it is and where to find it. Change it once, everything that references it updates automatically.

This is a team design decision — there's no single right answer. BearBots uses inner classes as a middle ground. As the codebase grows, teams often split into separate files: ArmConstants.java, FieldConstants.java, VisionConstants.java. Start here, split when it gets crowded. The pattern is more important than the file count.

Student
So Robot.java creates RobotContainer, and RobotContainer creates subsystems?
Exactly right
Robot.java creates one RobotContainer in its constructor. RobotContainer creates all the subsystems. Then every 20ms, Robot.java calls robotPeriodic(), which runs the CommandScheduler, which calls periodic() on every subsystem. That's the whole loop.
Student
What's LoggedRobot vs TimedRobot? The template gives me TimedRobot.
Good catch
LoggedRobot extends TimedRobot — it wraps each loop cycle with AdvantageKit logging. When we add AdvantageKit we change extends TimedRobot to extends LoggedRobot. All lifecycle methods work the same. You just get logging for free.
Student
Why does the arrow curve back on the left side of the loop box?
That's the loop!
That arrow is showing that after step 4, the robot doesn't stop — it immediately starts over at step 1. Every 20ms. Forever. Until you disable it or pull the battery. The robot is basically a very fast, very obedient hamster wheel.
Student
What happens if our code takes longer than 20ms to run?
Loop overrun — and it's bad
WPILib schedules robotPeriodic() every 20ms like clockwork. If your code takes 25ms, the next cycle starts 5ms late. Now your sensor readings are stale, your motor updates are delayed, and your control loop is drifting. Do it enough and the Driver Station prints a loop overrun warning. The fix: never put Thread.sleep(), heavy file I/O, or slow network calls inside periodic methods. Keep the loop lean.
Student
Does autonomous use completely different code from teleop?
Nope — same commands, different trigger
The exact same command classes you bind to buttons in teleop can run during autonomous. The only difference is what schedules them. In teleop a button press does it. In auto, autonomousInit() schedules one top-level command that kicks everything off. The subsystems, the IO layer, the scheduler — all identical.

Can I put everything in Robot.java? Technically yes. It works for about two weeks. Then you have 800 lines, two programmers editing the same file, and merge conflicts that make everyone cry.

How to declare a constant

Every constant follows the same three-word recipe: public static final.

public static final double kSpeedLimit  = 0.8;
public static final int    kLeftMotorPort = 0;
public static final String kProjectName = "BearBots";

Why all three?

  • public — any file in the project can read it
  • static — belongs to the class itself, not an object; no new DriveConstants() needed
  • final — can never be reassigned after startup; the value is locked
Why not the other keywords?
  • private instead of public — the constant becomes invisible to other files. Defeats the purpose.
  • No static — you'd need a new DriveConstants() object to read it. Never what you want.
  • No final — the value can be overwritten at runtime. It becomes a variable, not a constant.

Each subsystem gets its own constants file. Start with the files you need and add more as the project grows:

// DriveConstants.java
public final class DriveConstants {
    public static final double kSpeedLimit  = 0.8;
    public static final double kDeadband    = 0.1;
    public static final int    kLeftMotorPort  = 0;
    public static final int    kRightMotorPort = 1;
}

// ArmConstants.java
public final class ArmConstants {
    public static final double kMaxAngleDeg    = 120.0;
    public static final double kStowedAngleDeg = 5.0;
}

// AutoConstants.java
public final class AutoConstants {
    public static final double kDriveTimeSeconds = 2.0;
    public static final double kTurnTimeSeconds  = 1.3;
}
How to use a constant in another file

Add an import at the top of the file, then use the constant by name.

// At the top of Drive.java — import the constants file you need
import frc.robot.DriveConstants;

// Then use it anywhere in the file
xaxisSpeed = MathUtil.clamp(xaxisSpeed, -1.0, 1.0) * DriveConstants.kSpeedLimit;

No import? Use the fully-qualified name instead:

xaxisSpeed = MathUtil.clamp(xaxisSpeed, -1.0, 1.0) * DriveConstants.kSpeedLimit;
VS Code quick-fix import (Ctrl+.)

Type a constant class name that hasn't been imported yet — VS Code underlines it red. Hover it and press Ctrl+. then choose Add import. VS Code writes the import line for you.

How to name constants — the k rule

All WPILib constants start with a lowercase k, followed by UpperCamelCase. This is the convention used everywhere in WPILib's own source code — use it so your code looks familiar to every FRC programmer who reads it.

Category Examples
Speeds / power kSpeedLimit   kMaxTurnSpeed   kAutodriveSpeed
Angles kMaxAngleDeg   kStowedAngleDeg   kScoringAngleDeg
Distances kWheelCircumferenceMeters   kTrackWidthMeters   kWheelRadiusMeters
Timing kDriveTimeSeconds   kTurnTimeSeconds   kWaitTimeSeconds
Ports / IDs kLeftMotorPort   kRightMotorPort   kControllerPort
Strings / names kProjectName   kAutoDefault   kLogPrefix
Booleans kIsRightInverted   kEnableLogging   kIsSimulation

Rules worth memorising:

  • Always start with k — speedLimit ✗   kSpeedLimit ✓
  • Put the unit in the name when it could be ambiguous — kMaxAngle ✗   kMaxAngleDeg ✓
  • No abbreviations unless obvious — kWC ✗   kWheelCircumferenceMeters ✓
  • Drop the class name from the constant — DriveConstants.kDriveSpeed ✗   DriveConstants.kSpeed ✓
  • Booleans: use kIs or kEnable prefix — kMotorInverted ✗   kIsRightMotorInverted ✓
The unit rule prevents real bugs

WPILib uses it everywhere: kMaxVelocityMetersPerSecond, kWheelRadiusMeters. A function expecting radians and receiving degrees gives you a robot that spins 57× further than intended. The unit in the name is a free sanity check every time you read the code.

The comments even label them correctly. The code disagrees with the comments. The comments are right.

On a standard joystick: Y axis = forward/backward, X axis = left/right. Forward input should come from getLeftY(), turn from getLeftX(). They're swapped here.

Bug 3 symptom — Constants.java

The slightest joystick movement sends the robot to full speed instantly. There's no proportional control — it's either stopped or full throttle.

1public static final class Drive {

Bug 3 — kMaxOutput set to 100.0 instead of 1.0

100% sounds more powerful. It is not. Motor output is clamped to [-1, 1]. Everything above 1.0 is just 1.0 with extra steps.

Motor output values range from -1.0 (full reverse) to 1.0 (full forward). Any value multiplied by 100.0 immediately saturates the clamp. The motor sees 1.0 the instant the joystick moves at all.

Fix
public static final double kMaxOutput = 1.0;
Bug 4 symptom — DriveSubsystem.java

Driving "forward" makes the robot spin in place. The left encoder value goes negative while the right goes positive. Both motors are running but fighting each other.

1private final XRPMotor leftMotor = new XRPMotor(1);

3

Bug 4 — Left motor not inverted

The motors face opposite directions on the chassis. Running them the same way is not cooperation. It's a disagreement with wheels.

On a differential drive, the left and right motors are mechanically mirrored. To make both wheels push forward, one motor must be inverted. Here only rightMotor is inverted — left runs backwards relative to right, producing a spin instead of straight motion.

Fix
leftMotor.setInverted(true);
rightMotor.setInverted(false);
